XML Extensible Markup Language

Das erste Donnergrollen kam von den XML Entwicklern über ihre Pläne SGML zu vereinfachen. 
Diese Leute sagten, sie brauchten eine einfache Form von SGML 
die über das Web geliefert werden könnte. "SGML ist zu komplex für solch eine Aufgabe",
sagten sie, "und diePerformance würde ein Problem sein
außer wir könnten einige der unangenehmen Eigenschaften von SGML abstreifen".
Ihre Vision war die Leute dazu zu bringen, SGML weiter zu benutzen für komplexe Dokumente,
dann diese auszuliefern im Ganzen als XML über das Web. 
Die erklärte Absicht war nicht, SGML zu ersetzen, aber es zu erweitern.

Brian E. Travis

XML Extensible Markup Language

  • Elemente müssen geschlossen werden <p></p>
  • Leere Elemente müssen geschlossen werden <p />
  • Attribute sind Case Sensitive <Br /> != <br />
  • Attributwerte zwingend in Anführungszeichen Count=1
  • XML Deklaration muss vorhanden sein
  • DTD ist optional

SOAP Simple Object Access Protokoll

  • Untermenge von XML
  • Envelope und Body zwingend
  • Head optional
  • Envelope Namespace http://www.w3.org/2001/12/soap-envelope
  • Qualified Namespaced Elemente im Body

SOAP Simple Object Access Protokoll

                                            
<?xml version="1.0"?>
<soap:Envelope
  xmlns:soap="http://www.w3.org/2001/12/soap-envelope"
  soap:encodingStyle="http://www.w3.org/2001/12/soap-encoding">
    <soap:Body>
      <m:GetPrice xmlns:m="http://www.xyz.com/buecher">
        <m:Item>Apples</m:Item>
      </m:GetPrice>
    </soap:Body>
</soap:Envelope>
http://www.w3.org/TR/soap/

stateful / stateless

Unter einer Stateful Verbindung versteht man, dass eine Verbindung zum Server ständig aufrecht erhalten wird, sodass der Server den Client ebenfalls erreichen kann.


Unter einer Stateless Verbindung versteht man, dass eine Verbindung zu einem Server nur für eine Aufgabe aufgebaut, und danach sofort wieder beendet wird.


Was ist mit HTTP?

sync / async ( siehe AJAX )

Ein synchrones Laden einzelner Resourcen sagt aus, dass eine Resource immer erst vollständig geladen wird, bevor mit dem Script fortgefahren wird.


Ein asynchrones Laden einzelner Resourcen sagt aus, dass die Resourcen parallel geladen werden.


Achtung! Hierbei "synchron" != Gleichzeitig

REST

Die Kommunikation erfolgt auf Abruf. Der Client ist aktiv und fordert vom passiven Server eine Repräsentation an, bzw. modifiziert eine Resource. Stateless


Resourcen,besitzen eine ihnen zugeordnete URI, mit der sie adressiert werden können.


Die Representation einer Resource kann als Dokument vom Client angefordert werden.Puuuuuhhhh...Dokument?

REST

Repräsentationen können auf weitere Resourcen verweisen, die ihrerseits wieder Repräsentationen liefern, die wiederum auf Resourcen verweisen können.


Der Server verfolgt keinen Clientstatus. Jede Anfrage an den Server muss alle Informationen beinhalten, die zum Interpretieren der Anfrage notwendig sind.


Caches werden unterstützt. Der Server kann seine Antwort als Cache fähig oder nicht Cache fähig kennzeichnen.
