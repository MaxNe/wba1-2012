XML Extensible Markup Language

Das erste Donnergrollen kam von den XML Entwicklern über ihre Pläne SGML zu vereinfachen. 
Diese Leute sagten, sie brauchten eine einfache Form von SGML 
die über das Web geliefert werden könnte. "SGML ist zu komplex für solch eine Aufgabe",
sagten sie, "und diePerformance würde ein Problem sein
außer wir könnten einige der unangenehmen Eigenschaften von SGML abstreifen".
Ihre Vision war die Leute dazu zu bringen, SGML weiter zu benutzen für komplexe Dokumente,
dann diese auszuliefern im Ganzen als XML über das Web. 
Die erklärte Absicht war nicht, SGML zu ersetzen, aber es zu erweitern.

Brian E. Travis

XML Extensible Markup Language

XML ist eine Untermenge von SGML

XML Extensible Markup Language

  • Elemente müssen geschlossen werden <p></p>
  • Leere Elemente müssen geschlossen werden <p />
  • Namen von Elementen unterliegen folgenden Regeln:
    • düfen keine Sonderzeichen enthalten
    • dürfen nicht mit Zahlen anfangen
    • dürfen nicht mit "XML" anfangen
    • dürfen keinen Whitespace enthalten

XML Extensible Markup Language

  • Elemente und Attribute sind Case Sensitive <Br /> != <br />
  • Attributwerte zwingend in Anführungszeichen Count=1
  • XML Deklaration muss vorhanden sein
  • DTD ist optional ( Validität )

SOAP Simple Object Access Protokoll

  • Datenformat durch XML festgelegt
  • Envelope und Body zwingend
  • Head optional
  • Envelope Namespace http://www.w3.org/2001/12/soap-envelope
  • Qualified Namespaced Elemente im Body

SOAP Simple Object Access Protokoll

<?xml version="1.0"?>
<soap:Envelope
  xmlns:soap="http://www.w3.org/2001/12/soap-envelope"
  soap:encodingStyle="http://www.w3.org/2001/12/soap-encoding">
    <soap:Body>
      <m:GetPrice xmlns:m="http://www.xyz.com/buecher">
        <m:Item>Apples</m:Item>
      </m:GetPrice>
    </soap:Body>
</soap:Envelope>
http://www.w3.org/TR/soap/

JSON JavaScript Object Notation

Die Javascript Object Notation ist ein Datenaustausch-Format, um Daten einfach über das Web auszutauschen. Die Sprache ist in ECMA Script und durch das W3C definiert und ist zu JavaScript als nativ anzusehen. Sie ist eines der wichtigsten Datenaustauschformate im WEB 2.0!

JSON JavaScript Object Notation

Warum JSON und nicht XML?

  • Wird von JS als natives Object angesehen ( mit eval() oder JSON.parse() )
  • ABER VORSICHT! eval() ist evil ;-)
  • Weniger Overhead als XML
  • Menschenlesbar
http://www.ietf.org/rfc/rfc4627.txt

JSON JavaScript Object Notation

JSON Datentypen

  • Object = {}
  • Array = []
  • Number = 0-9.
  • String = ""
  • Boolean = true false
  • Null = null

Mime-Type: application/json

JSON Beispiel

{
	"empty":null,
	"boolean":false,
	"number":5.2,
	"string":"Hallo Welt",
	"Object":{"Key":Value},
	"Array":[Element1, Element2],
	"funky":function(foo){
		alert("Hello World, "+foo);
	}
}

stateful / stateless

Unter einer Stateful Verbindung versteht man, dass eine Verbindung zum Server ständig aufrecht erhalten wird, sodass der Server den Client ebenfalls erreichen kann bzw. dass der Server den Status des Clients speichert.


Unter einer Stateless Verbindung versteht man, dass eine Verbindung zu einem Server nur für eine Aufgabe / Request aufgebaut, und danach sofort wieder beendet wird.

REST / CRUD

REST: Representational State Transfer

Erstmals von Roy Fielding in seiner Dissertation beschrieben.

Kein W3C oder sonstiger Standard

REST ist ein Paradigma um Webanwendungen zu erstellen


CRUD: Create,Read,Update,Delete

REST

Die Kommunikation erfolgt auf Abruf. Der Client ist aktiv und fordert vom passiven Server eine Repräsentation an, bzw. modifiziert eine Resource. Stateless


Resourcen,besitzen eine ihnen zugeordnete URI, mit der sie eindeutig adressiert werden können.


Eine Representation einer Resource kann als Dokument vom Client angefordert werden. Eine Resource kann mehrere Repräsentationen haben

REST

Repräsentationen können auf weitere Resourcen verweisen, die ihrerseits wieder Repräsentationen liefern, die wiederum auf Resourcen verweisen können. Stichwort Hypermedia


Der Server verfolgt keinen Clientstatus. Jede Anfrage an den Server muss alle Informationen beinhalten, die zum Interpretieren der Anfrage notwendig sind.


Caches werden unterstützt. Der Server kann seine Antwort als Cache fähig oder nicht Cache fähig kennzeichnen.

GET

Zum Lesen einer Resource

Maximale Länge begrenzt ( Browser, nicht HTTP )

Sichtbarkeit in Addressleiste

Cacheable


http://tools.ietf.org/html/rfc2616#section-9.3

POST

Zum Erstellen einer Sub-Resource

Nicht zwangsläufig neue Resource

Wenn eine neue Resource, dann URI vom Server

Länge ist theoretisch unbegrenzt


http://tools.ietf.org/html/rfc2616#section-9.5

PUT

Zum ändern oder Anlegen einer Resource

URI kommt vom Client, nicht vom Server

Kann Seiteneffekte haben


http://tools.ietf.org/html/rfc2616#section-9.6

DELETE

Zum Löschen einer Resource

Kann Seiteneffekte haben


http://tools.ietf.org/html/rfc2616#section-9.7

HEAD

Mit der GET Methode identisch

Es wird kein Response-Body mitgesendet


http://tools.ietf.org/html/rfc2616#section-9.4
